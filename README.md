# LLM Thinking Effort Visualizer

> Can you tell how hard a question is *before* an LLM answers it?  
> This project builds the instrumentation to find out — and reports what the data actually shows.

---

## The Idea

We've been conditioned by Google to expect instant answers. LLMs inherit this: a trivial question and a genuinely hard one both return in roughly the same time, flattening our perception of difficulty. This project asks: **can we build a system that surfaces real inference effort — and can we predict that effort from the prompt alone?**

The answer turns out to be more interesting than "yes" or "no."

---

## What's in the Repo

| File | What it does |
|---|---|
| `thinking_effort_ollama.html` | Browser tool — open locally, talks to Ollama, no server needed |
| `train_predictor_ollama.py` | Collects ground-truth latency data and fits a linear predictor |
| `prompts.json` | 55-prompt dataset spanning trivial → deceptive → genuinely hard |
| `results_ollama.json` | Labeled dataset (generated by running the pipeline) |
| `weights_ollama.json` | Fitted predictor weights (paste into browser tool) |

## How It Works

### Signals (what we can measure without reasoning traces)

Running against **Mistral via Ollama** — a local model with no exposed thinking trace — we measure real signals from the Ollama API:

- **`eval_duration`** — actual generation time in nanoseconds, converted to ms
- **`eval_count` / `eval_duration`** — tokens per second (drops on complex generation)
- **`eval_count`** — output token count (proxy for answer depth)
- **Cognitive churn** — count of backtracking phrases in the output text ("wait", "actually", "let me reconsider", ...) normalized per 100 output tokens

These are honest signals. No fake delays, no theater.

### Pre-Inference Predictor

Before each call, a lightweight linear model predicts expected generation time from 8 surface features of the prompt:

| Feature | Intuition |
|---|---|
| `log(word_count)` | Longer prompts → more context to process |
| `question_mark_count` | Multi-question prompts → longer answers |
| `logic_word_count` | "prove", "derive", "explain why" → harder |
| `constraint_word_count` | "must", "only", "exclude" → constrained reasoning |
| `multipart_score` | Numbered sub-questions, multiple conjunctions |
| `ambiguity_markers` | "could", "depends", "arguably" → hedged answers |
| `code_markers` | Code in prompt → code in answer → longer gen |
| `digit_count` | Numbers → math → longer chains |

The predictor runs in-browser in milliseconds before the API call starts. The bar scales against this prediction: if the model blows past it, the bar turns red and switches to "overtime" mode.

### Deceptive Simplicity Detector

Prompts where predicted gen time is low but actual is high get flagged automatically. These are the interesting cases — questions that *look* trivial but triggered unexpectedly long or tortured responses.

## Empirical Results

### R² is the finding, not a failure metric

Running the full 55-prompt dataset through Mistral:

- **Cross-validated R² on generation time: approximately 0.1–0.3**  
  Surface prompt features are weak predictors of how long Mistral takes to respond. The dominant driver is output length — and output length is driven by how verbose Mistral decides to be, not purely by question difficulty.

- **Churn is non-zero on hard prompts**  
  On trivial factual questions, churn is 0.0 — Mistral doesn't backtrack on things it knows. On multi-step reasoning, constraint problems, and deceptive puzzles, pivot phrases appear and the churn score rises.

- **The scatter plot reveals structure**  
  Most prompts cluster near the diagonal. Deceptive outliers — predicted fast, actually slow — tend to be trick questions, self-referential prompts, or problems that require re-reading the question before committing.

### What this means

**Reasoning depth in LLMs without explicit chain-of-thought is not reliably readable from prompt surface features.** The effort happens inside a single forward pass, not in a way that maps cleanly onto syntactic prompt properties. Models with exposed reasoning traces (Claude extended thinking, DeepSeek-R1) give a much stronger signal — reasoning token count is direct ground truth. With a standard generation model, you're reading shadows on the wall.

That's an interesting result. It motivates why exposed thinking traces matter as a transparency primitive.

## Running It

### Prerequisites

```bash
# Ollama running with mistral pulled
ollama serve
ollama pull mistral

# Python
pip install scikit-learn
```

### 1. Open the browser tool

Open `thinking_effort_ollama.html` in any browser. No server needed — it calls `localhost:11434` directly.

Hit **"test connection"** first. Then ask questions and watch the bar fill in real time as tokens stream back. Try trivial vs. hard prompts back to back to see the latency and churn difference.

### 2. Collect the training dataset

```bash
# Sanity check on 10 prompts first
python3 train_predictor_ollama.py \
  --prompts prompts.json \
  --out results_ollama.json \
  --weights weights_ollama.json \
  --limit 10

# Full run (~55 prompts, 5–15 minutes depending on hardware)
python3 train_predictor_ollama.py \
  --prompts prompts.json \
  --out results_ollama.json \
  --weights weights_ollama.json

# Resume if interrupted
python3 train_predictor_ollama.py \
  --prompts prompts.json \
  --out results_ollama.json \
  --weights weights_ollama.json \
  --resume
```

The console will print cross-validated R², deceptive-simplicity candidates, and highest-churn prompts.

### 3. Load fitted weights into the browser

Copy the contents of `weights_ollama.json`, expand **"predictor weights (advanced)"** in the browser tool, paste, and click **"apply weights"**. The bar now uses your fitted model.

### 4. Try a different model

```bash
python3 train_predictor_ollama.py \
  --model gemma3:1b \
  --prompts prompts.json \
  --out results_gemma.json \
  --weights weights_gemma.json
```

Change the model field in the browser tool to `gemma3:1b`. Comparing R² across models is itself a useful experiment.

## Extending the Dataset

`prompts.json` is a plain JSON array. Add your own entries:

```json
{"prompt": "your question here", "category": "your-label"}
```

More prompts in the hard and deceptive tiers improve the regression and give the scatter plot more structure.

---

## Architecture Notes

Feature extraction in `train_predictor_ollama.py` and `thinking_effort_ollama.html` are kept in sync — same keyword lists, same math. Weights fitted by the Python script transfer directly into the browser with no conversion.

The predictor is Ridge regression (`alpha=1.0`). Intentionally simple — the goal is to test whether the signal exists at all, not to maximize accuracy with a complex model.

## Potential Extensions

- **Reasoning model comparison** — run the same prompts through a model with exposed thinking traces (Claude API with `thinking: enabled`, or DeepSeek-R1 locally). Compare R² on reasoning token count vs. generation time. The gap is the empirical cost of not having a thinking trace.
- **Per-model weight comparison** — fit separate weights for Mistral vs. Gemma and compare coefficients. Do the same surface features matter across architectures?
- **Live churn streaming** — currently churn is computed post-hoc. Could score it as tokens arrive and update the UI in real time.
- **Feature ablation** — systematically vary a single prompt (add constraints, sub-questions, ambiguity) and measure the effect on gen_ms. Empirically validates or falsifies individual feature coefficients.
